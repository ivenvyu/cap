
# 전체 설계

## 계층적·다중적 관점의 이미지 태깅 온톨로지 설계

```bash

“안개 낀 풍설기천년”과 같은 복합적 이미지 검색을 지원하기 위해서는 단순한 공간명 기반 태깅만으로는 충분하지 않다. 

검색 가능성을 높이기 위해서는 메타데이터를 동시에 관리하는 계층적 온톨로지 설계가 필요하다.

메타데이터는 다음과 같이 구분할 수 있다.

- 공간(Spatial) : 정원, 건축물, 산책로, 실내, 실외
- 시간(Temporal) : 계절, 시간대, 날씨
- 물체(Object) : 식물, 조형물
- 무드(Mood) : 정적, 고요함, 활기, 웅장함
- 활용 목적(Usage) : SNS, brochure, 제안서, 배너

이러한 체계는 실제 이미지 데이터 분포를 분석하고, 이미지 임베딩 기반 실험을 통해 검색 성능을 검증하면서 점진적으로 최적화되어야 한다.

즉, 검색 품질을 극대화할 수 있는 메타데이터 구조를 데이터 기반으로 설계해야 한다.

```

## 대용량 고해상도 이미지 처리 파이프라인 설계

```
수천~수만 장의 고해상도 이미지를 임베딩하고 태깅하는 과정에서는 GPU 비용 관리가 매우 중요하다. 따라서 검색 품질을 유지하면서도 비용 효율적인 처리 전략 이 필요하다.
- Manifest를 생성하여 시스템이 이미지 데이터들을 추적하고 처리할 수 있게 만드는 일종의 데이터셋 관리 장부이다.
- Thumbnail로 원본 이미지의 픽셀을 작게 줄인 복사본을 생성한다면 GPU 처리 비용을 절감할 수 있다.
(ex. A설정 :
        thumbnail 521px
        batch_size 32
        1000장 처리 속도 90초

     B설정 :
        thumbnail 768px
        batch_size 16
        1000정 처리 속도 180초

B의 검색 품질이 A보다 크게 좋지 않는 이상 A를 선택하는 것이 합리적이다. 즉, 검색 품질 손실 없이 가장 비용 효율적인 설정을 찾는 것이며
실제 데이터셋을 기반으로 직접 실험하여 측정해야 하는 empirical optimization 문제이다.



```

## CLIP 기반 태깅, 클러스터링
```bash

CLIP embedding 기반 clustering은 이미지 전체를 사람이 처음부터 하나씩 태깅하는 부담을 줄이고, **유사한 이미지를 묶어 검토 단위로 만드는 과정**이다. 즉, 개별 이미지 단위의 수작업 라벨링이 아니라, 먼저 모델이 이미지의 의미적 유사도를 기준으로 군집을 만들고, 사람은 그 군집이 실제 검색 목적에 맞게 잘 구성되었는지를 검토한다.

먼저 각 이미지를 CLIP encoder에 입력하여 이미지 embedding을 생성한다. 이 embedding은 이미지의 시각적 특징뿐 아니라 텍스트 질의와 연결될 수 있는 의미적 특징을 포함한다. 이후 cosine similarity 또는 embedding distance를 기준으로 K-means, HDBSCAN, hierarchical clustering 등의 방법을 적용하여 유사 이미지 그룹을 만든다.

예를 들어 다음과 같은 cluster가 생성될 수 있다.

| Cluster | 모델이 묶은 이미지 특성 | 사람이 검토하며 보정할 라벨 |
|----------|--------------------------|------------------------------|
| Cluster A | 눈 덮인 정원 이미지 | 겨울, 설경, 정원, 정적, 브로슈어 |
| Cluster B | 안개 낀 산책로 이미지 | 안개, 산책로, 고요함, 아침, SNS |
| Cluster C | 전통 건축물과 정원 이미지 | 건축물, 정원, 전통성, 제안서 |
| Cluster D | 방문객이 포함된 야외 이미지 | 방문객, 활기, 체험, 홍보 콘텐츠 |

하지만 같은 cluster 안에 있다고 해서 모든 이미지에 동일한 라벨을 부여할 수 있는 것은 아니다. 예를 들어 “눈 덮인 정원” cluster 안에도 어떤 이미지는 건축물이 중심이고, 어떤 이미지는 식물이 중심이며, 어떤 이미지는 방문객이 포함되어 있을 수 있다. 따라서 사람은 cluster 단위로 빠르게 검토하되, 필요한 경우 세부 라벨을 이미지 단위로 보정한다.

이때 중요한 점은 사람이 단순히 라벨을 붙이는 것이 아니라, **모델이 잘못 묶은 이유와 검색 실패 가능성을 함께 기록하는 것**이다. 예를 들어 다음과 같은 피드백이 누적될 수 있다.

| 문제 유형 | 예시 | 개선 방향 |
|-----------|------|------------|
| 과도한 묶임 | 눈이 있는 이미지를 모두 같은 cluster로 묶음 | 공간/피사체 기준 세분화 |
| 과도한 분리 | 같은 정원인데 조명 차이로 다른 cluster로 분리 | 장소명/구조물 라벨 강화 |
| 검색 의도 미반영 | “고요한 겨울 정원” 검색에 적합한 이미지가 누락됨 | 무드 라벨 추가 |
| 활용 목적 불명확 | SNS용 이미지와 제안서용 이미지가 섞임 | usage 라벨 별도 관리 |

이렇게 수정된 라벨 데이터는 Manifest에 누적된다. Manifest에는 단순한 파일 경로뿐 아니라 이미지 ID, 원본 경로, 썸네일 경로, embedding 버전, cluster ID, 자동 생성 라벨, 사람이 보정한 라벨, 검수 상태, 검색 성능 평가 결과 등이 기록된다.

예시는 다음과 같다.

| image_id | cluster_id | auto_tags | human_tags | status |
|-----------|------------|------------|-------------|---------|
| img_001 | winter_03 | winter, garden, snow | winter, snow, garden, quiet, brochure | verified |
| img_002 | winter_03 | winter, garden, snow | winter, architecture, snow, proposal | corrected |
| img_003 | fog_02 | fog, path | fog, morning, path, calm, SNS | verified |

이 데이터가 누적되면 두 가지 방향으로 개선이 가능하다.

### 1. Tagging model 개선

사람이 보정한 라벨을 학습 데이터로 사용하여, 이후 신규 이미지에 대해 더 정확한 자동 태깅을 수행할 수 있다. 예를 들어 초기에는 CLIP 유사도 기반으로 “겨울, 정원” 정도만 예측하던 시스템이, 반복 학습 후에는 “겨울, 설경, 정적, 브로슈어 적합”처럼 더 세밀한 태그를 제안할 수 있다.

### 2. Ontology 개선

실제 검수 과정에서 자주 등장하지만 기존 체계에 없던 개념을 새로운 태그 또는 상위 카테고리로 반영한다. 예를 들어 “고요함”, “몽환적”, “전통성”, “프리미엄”, “체험형” 같은 검색어가 반복적으로 필요하다면, 이를 mood 또는 usage 축의 정식 태그로 편입한다.

결과적으로 이 프로세스는 다음과 같은 순환 구조를 갖는다.

1. 이미지에서 CLIP embedding 생성  
2. embedding 기반 clustering 수행  
3. cluster 단위로 사람이 빠르게 검토  
4. 이미지별 세부 라벨 보정  
5. 수정된 라벨과 오류 유형을 Manifest에 기록  
6. 누적 데이터를 기반으로 tagging model 또는 ontology 업데이트  
7. 신규 이미지에 개선된 모델과 태그 체계 적용  
8. 검색 결과 품질을 평가하고 다시 보정  

이 반복 구조의 핵심은 **검색 실패 사례를 데이터로 전환하는 것**이다. 사용자가 원하는 이미지를 찾지 못했거나, 검색 결과 상위에 부적절한 이미지가 노출된 경우 이를 단순 오류로 처리하지 않고, “어떤 라벨이 부족했는지”, “어떤 cluster가 너무 넓었는지”, “어떤 메타데이터 축이 새로 필요한지”를 분석한다.

따라서 시간이 지날수록 시스템은 단순히 이미지 수가 늘어나는 것이 아니라, **검색 의도에 맞게 정교화된 이미지 지식베이스**로 발전한다. 이 방식은 초기 구축 비용을 줄이면서도, 실제 사용자 검색 로그와 검수 피드백을 반영해 점진적으로 검색 품질을 개선할 수 있다는 장점이 있다.

```
